# Re:cord Current Truth

기준 날짜: 2026-03-27  
작성 목적: 지금 이 repo 기준으로 무엇이 핵심 제품인지, 어디까지 준비되었는지, 무엇을 우선해야 하는지 흔들리지 않게 정리하는 기준 문서

---

## 1. 한 줄 정의

Re:cord는 `상담 전에 보내는 공개 신뢰 링크`다.

더 구체적으로 말하면:

- 흩어진 후기
- 직접 받은 후기
- 작업 사진
- 간단한 소개
- 상담/예약 버튼

을 한 공개 페이지에 모아, 고객이 사람의 실력과 분위기를 먼저 이해하게 만드는 제품이다.

---

## 2. 지금 우리가 실제로 파는 것

현재 Re:cord의 중심 가치는 아래다.

1. 리뷰를 많이 모으는 도구가 아니다.
2. OCR 정확도를 자랑하는 도구가 아니다.
3. 대시보드를 오래 쓰게 만드는 도구가 아니다.
4. `보내는 링크 1장`으로 상담 전 신뢰를 만드는 도구다.

지금 제품의 가장 중요한 질문은:

`이 링크를 고객에게 보냈을 때, 30초 안에 누구인지 / 왜 믿을 수 있는지 / 어디로 문의하면 되는지가 전달되는가?`

---

## 3. 현재 타깃

현재 가장 선명한 wedge는 `헤어디자이너`다.

이유:

- 샵 이동, 독립, 1인샵 오픈처럼 신뢰 단절 순간이 분명하다.
- 고객이 샵보다 사람을 따라 움직이는 비중이 크다.
- 후기, 작업 사진, 소개 링크가 상담 전환과 바로 연결된다.
- 카카오톡, 인스타, DM, 공지로 링크를 보내는 장면이 이미 존재한다.

확장 후보는 있다.

- PT 트레이너
- 보컬/레슨
- 네일/뷰티
- 크몽 프리랜서

하지만 현재 제품 우선순위는 `헤어디자이너 first`다.

동시에, 라이브 `syb2020` 프로필은 제거 대상이 아니라 `founder의 실사용 proof profile`로 유지한다.

즉 현재 기준은:

- go-to-market wedge는 헤어디자이너
- founder proof asset은 `syb2020`

이다.

---

## 4. 지금 기준 제품 핵심 흐름

현재 가장 자연스러운 메인 플로우는 아래다.

1. 사용자명과 공개 프로필을 만든다.
2. 대표 후기와 작업 사진을 정리한다.
3. 상담 버튼을 연결한다.
4. 공개 상태를 켠다.
5. 링크를 복사해서 카톡, 문자, DM, 인스타 프로필 등에 보낸다.
6. 이후에는 직접 후기 요청 링크로 새 후기를 계속 쌓는다.

이 흐름에서 중요한 건 입력량이 아니라 `대표 증거를 먼저 보여주는 것`이다.

---

## 5. 지금 실제로 성숙한 영역

아래는 현재 기준으로 제품 코어에 해당한다.

### A. 공개 프로필

- 공개 프로필 로딩
- 공개/비공개 제어
- 원문 링크, 이미지, 직접 후기 구분
- 공개용 payload 조합
- 대시보드 미리보기와 공개 페이지가 같은 로직을 공유

이건 현재 제품의 가장 성숙한 코어다.

### B. 리뷰 관리

- 리뷰 생성/수정/삭제
- 공개 여부 관리
- 대표 증거 후보 정리
- 직접 받은 후기 검토
- 요청 링크 복사

현재 실제 작업 공간의 중심은 이 영역이다.

### C. 프로필 편집

- 이름, 소개, 위치
- 상담 링크/전화
- 포트폴리오 이미지
- 공개 상태
- 공개 미리보기

사용자가 “바로 보낼 링크”를 만드는 데 필요한 핵심은 이미 갖춰져 있다.

### D. 공유

- 링크 복사
- 소개 멘트 복사
- 카카오 공유
- 인스타/스레드용 복사
- QR 생성

Re:cord의 현재 제품 정체성상 이 기능은 부가 기능이 아니라 핵심 기능이다.

### E. 직접 후기 수집

- 공개 후기 요청 페이지
- 제출
- pending 저장
- 소유자 검토 후 공개

이 흐름은 “과거 후기 보관” 이후의 미래 자산 축적을 담당한다.

## 6. 지금 살아 있지만 메인 제품이 아닌 영역

아래는 코드에 존재하고 의미도 있지만, 현재 메인 제품의 정체성으로 내세우면 안 되는 영역이다.

### A. OCR

- 여전히 큰 코드 영역이다.
- fixture 테스트도 존재한다.
- bulk upload와 함께 강한 온보딩 수단이다.

하지만 지금 기준으로 OCR은 `메인 메시지`가 아니라 `입력 보조 수단`이다.

### B. Bulk Upload

- CSV 일괄 업로드
- OCR 스크린샷 대량 처리

파워유저나 운영팀 입장에서는 의미가 크다.  
하지만 메인 홈/대시보드의 주인공이 되어서는 안 된다.

### C. Migration Request / Concierge Setup

- 지금은 실제로 꽤 중요한 전환 도구다.
- 베타 고객에게 자료를 받아 초안을 빨리 열어주는 역할을 한다.

다만 이것은 `제품 정체성`이라기보다 `초기 운영 방식`에 가깝다.

### D. Reboot / Customize / 과거형 확장축

- repo 안에는 여전히 남아 있다.
- 일부는 실험적이거나 옛 전략의 흔적이다.

현재 기준에서는 코어가 아니다.

## 7. 지금 기준으로 혼선이 나는 이유

현재 repo에는 세 가지 층위가 동시에 존재한다.

### 1. 새 제품 서사

- 상담 전에 보내는 신뢰 링크
- 공개 프로필
- 대표 후기 3개
- 작업 사진
- 공유 행동 중심

### 2. 베타 운영 방식

- migration request
- concierge onboarding
- 스크린샷/영상/PDF 수동 수집
- 초기 세팅 도움

### 3. 예전 제품 DNA

- OCR 중심 설명
- 리뷰 아카이브 SaaS 톤
- 커스터마이징/통계/확장 기능 강조
- 여러 버티컬을 동시에 잡는 서사

즉, 지금 문제는 “방향이 없음”이 아니라 `여러 세대의 제품이 동시에 repo 안에 남아 있는 것`이다.

## 9. 운영 현실

현재 상태를 과장 없이 정리하면:

### 코드/로컬 품질

- `npm run verify` 통과
- `npm run smoke:local` 통과
- `npm run secrets:check` 통과

즉, 로컬 기준으로는 꽤 건강하다.

### 프로덕션 준비

- `NODE_ENV=production npm run preflight:prod`는 아직 blocker를 보여준다.
- 핵심 blocker는 운영 env, DB baseline, remote smoke, OAuth/Lemon 외부 설정이다.

즉:

- 제품 구현은 높다.
- 운영 readiness는 아직 미완이다.

---

## 10. 지금 가장 중요한 제품 원칙

아래는 앞으로 기능 우선순위를 결정할 때 지켜야 하는 기준이다.

1. 공유가 안 되면 제품이 아니다.
2. 대표 후기 3개가 후기 30개보다 중요하다.
3. 링크를 보내기 전까지의 마찰을 줄이는 게 우선이다.
4. OCR은 중요하지만 필수 정체성이 아니다.
5. concierge는 초기 운영 수단이지, 장기적인 제품 정의가 아니다.
6. 초기에는 헤어디자이너에 더 집중한다.
7. 기능 확장보다 제품 메시지 정렬이 먼저다.

## 11. 지금 당장 해야 하는 의사결정

가장 중요한 질문은 하나다.

`OCR / bulk upload / migration concierge를 앞으로도 메인 제품처럼 노출할 것인가, 아니면 내부 fulfillment layer로 내릴 것인가?`

이 질문에 답해야 아래가 함께 정리된다.

- README
- 가격 페이지 메시지
- 가이드/법률 문구
- IA
- 기능 노출 우선순위
- 스키마/결제/운영 정리 방향

현재 기준으로는 아래가 가장 자연스럽다.

- 공개 프로필 / 대표 후기 / 공유: 메인
- direct review request: 메인
- migration concierge: 베타 운영 레이어
- OCR / bulk upload: 보조 입력 레이어
