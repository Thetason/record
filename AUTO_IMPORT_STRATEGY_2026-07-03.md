# 리뷰 자동 이관 전략 — "네이버 주소 딸깍" 실현 방법

기준 날짜: 2026-07-03
성격: "스크린샷 수동 업로드"를 줄이는 방법에 대한 기술·법률 확정 판단. 다중 에이전트 리서치(법률/기술/공식API/반자동화) + 판례 실측 기반.

---

## 1. 한 줄 결론

**"우리 서버가 네이버를 크롤링해서 전부 긁어오는 딸깍"은 하면 안 된다** (기술적으로 깨지기 쉽고, 한국법상 실제 배상 판결이 난 영역). 하지만 **유저 체감을 "딸깍"에 근접시키는 합법 경로는 있다.** 핵심 원칙: **데이터에 접근하는 주체를 항상 "로그인한 유저 본인"에 두고, 우리 서버는 네이버에 요청하지 않는다.**

---

## 2. 왜 서버 크롤링은 안 되는가 — 결정적 이유

가장 중요한 포인트 (이걸 놓치면 전략이 무너진다):

> **리뷰를 쓴 사람은 사장님이 아니라 "손님"이다.**

그래서 "내 데이터 옮기기(data portability)"가 아니다. 세 겹의 권리가 전부 사장님 것이 아니다:

| 권리 | 권리자 | 사장님 동의로 해결되나? |
|---|---|---|
| 리뷰 텍스트 저작권 (창작성 있는 후기) | **작성한 손님** | ❌ |
| 리뷰 DB 제작자 권리 (저작권법 §93) | **플랫폼(네이버)** | ❌ |
| 작성자 개인정보(닉네임·식별자) 처리 동의 | **작성한 손님** | ❌ |

**사장님이 "내 가게 리뷰 가져와줘"라고 아무리 동의해도, 위 세 리스크 중 어느 것도 치유되지 않는다.**

### 실제 판례 (한국은 크롤링으로 돈 물린 나라다)

- **사람인 vs 잡코리아**: 경쟁사 채용정보 무단 크롤링·게재 → 저작권법 §93 DB제작자 권리 침해. **총 4.5억원 배상 확정** (대법 심리불속행 기각). [출처](https://www.lawtimes.co.kr/opinion/117765)
- **야놀자 vs 여기어때**: 형사(정보통신망법)는 **최종 무죄**(공개정보 + 기술적 보호조치 우회 없음). 그러나 **민사에서 부정경쟁방지법 (카)목 성과도용으로 10억원 배상 확정.** → **"형사 무죄 ≠ 적법."** [출처](https://www.lawtimes.co.kr/news/articleView.html?idxno=172326)
- **부정경쟁방지법 (카)목**: 저작권·DB권이 없어도 "상당한 투자로 만든 성과를 경쟁질서에 반해 무단 사용"하면 성립하는 보충 일반조항. 크롤링·데이터 재사용에 광범위 적용.

### 기술적으로도 취약

- 네이버 플레이스 리뷰는 내부 비공개 엔드포인트로 로드 + 봇 차단/토큰/약관(자동수집 금지).
- **네이버가 2026년 리뷰 UI를 대대적으로 개편 중**(5년 만에 별점 부활 등) → DOM/셀렉터 파싱은 계속 깨진다. 유료 고객에게 "됐다 안 됐다"는 신뢰 파괴.
- 공개된 사장님용 "리뷰 대량 export API"는 존재하지 않음(2026-07 확인).

## 3. 유저의 욕망에 가장 근접하는 합법 사다리

플랫폼별로 "가능한 최선"이 다르다. 하나의 만능 딸깍은 없고, 플랫폼별 최적 경로를 조합한다.

### 등급표 (법적안전 × 마찰감소 × 구현비용)

| 경로 | 법적 안전 | 유저 마찰 | 구현 비용 | 판정 |
|---|---|---|---|---|
| **A. 벌크 스크린샷/스크롤캡처 → 멀티모달 OCR** (네이버·카카오·당근) | 🟢 높음 (서버가 네이버 접근 0) | 🟡 캡처 1~2회 | 🟢 낮음 (절반 이미 구축) | **1순위 · 즉시** |
| **B. 구글 = 공식 Business Profile API** | 🟢 높음 (owner OAuth 정식) | 🟢 진짜 딸깍 | 🟡 중간 (API 승인 필요) | **2순위** |
| **C. 컨시어지 (사람이 대신 정리)** | 🟢 높음 | 🟢 유저는 자료만 전달 | 🟢 제로빌드 | **초기 브릿지** |
| **D. Re:cord 자체 후기 요청 링크** (작성자 동의 수집) | 🟢🟢 완전 안전 + 자산 | 🟡 시간 필요 | 🟢 이미 있음 | **장기 코어** |
| E. 브라우저 확장 (유저가 자기 계정서 export) | 🟡 중간 (네이버 약관 + Chrome 정책 2026.8.1 강화) | 🟢 낮음 | 🔴 높음·유지비 큼 | 보류 |
| F. 서버 크롤링 | 🔴 판례상 실질 리스크 | 🟢 낮음 | 🔴 깨지기 쉬움 | **금지** |
| G. 텍스트 붙여넣기/공유시트 | 🟢 높음 | 🔴 커버리지 낮음 | 🟢 낮음 | 보조 |

### 표시 방식도 재설계 (리스크 원천 차단)

크롤링을 안 하는 것만으론 부족하다. **"텍스트 전문 복제 저장"에서 다음으로 전환:**

1. 리뷰는 **요약/발췌** + **원문 딥링크**(재호스팅이 아니라 출처 표시)
2. 타 플랫폼 평점은 **집계 수치(별점 평균·건수 같은 사실정보)**만 표시 — 사실정보는 저작물이 아님
3. 작성자는 **익명화**(닉네임 노출 최소화)
4. "우리 리뷰"로 당당히 내세우는 건 **Re:cord 후기 요청으로 직접 받은 것**뿐

이건 저장소의 기존 `platform_review_legal_analysis.md` 결론(요약/발췌, 텍스트 출처만, 면책조항)과도 일치한다.

## 4. 이번 주에 뭘 만들까 (구체 실행)

**목표: "리뷰 30개를 하나씩 찍는" → "긴 화면 1~2장 올리면 AI가 통째로 분해"**

기존 `/dashboard/bulk-upload` + `/api/ocr`는 이미 다중 스크린샷 OCR + 플랫폼 감지 + 정제를 한다. 남은 건 **캡처 횟수를 줄이는 것.**

### Step 1 (구현 완료 — 2026-07-03, 미커밋) ✅
"긴 캡처 1장 → 리뷰 여러 개" 매직 플로우를 Claude 비전으로 구현했다. 유저 검수(신뢰도 배지) 스텝을 UX로 흡수해 정확도 부담을 신뢰 강화로 전환.

**플랫폼별 스마트 판독:** 비전 LLM은 화면을 의미로 읽어 형식이 다 달라도 근본 대응 가능. 여기에 플랫폼별 "해부 프로필"(`lib/review-platforms.ts`)을 주입해 각 플랫폼의 함정을 처리한다:
- 네이버(2026-04 별점 부활): 방문자/영수증/블로그 리뷰 구분, **키워드(태그 집계) 리뷰는 개별 리뷰 아님 → 제외**
- 당근: **거래후기는 별점 없음 → rating 지어내지 않고 null**(매너온도는 리뷰 별점 아님)
- 크몽: 항목별 별점·구매옵션, 작성자 마스킹("ho****") 유지
- 숨고: 고수 리뷰 5점 + 서비스 카테고리
- 인스타/카톡 DM: 상대 말풍선만(내 메시지 제외), 별점 없음
- 한 캡처에 여러 플랫폼이 섞여 있어도 리뷰마다 플랫폼 자동 판별. reviewType 라벨 부여.

착지 파일:
- `lib/review-platforms.ts` — 플랫폼 프로필(해부 지식·별점 유무·별칭 정규화·프롬프트 가이드 생성).
- `lib/claude-vision.ts` — Claude 비전 다중 리뷰 추출(`claude-opus-4-8` 기본, `ANTHROPIC_VISION_MODEL`로 `claude-haiku-4-5` 등 저비용 교체 가능). 키 없으면 dev는 mock·prod는 503.
- `app/api/ocr/multi/route.ts` — 이미지 1~5장 → 구조화 리뷰 배열(sharp로 정규화).
- `app/api/reviews/import/route.ts` — 확인된 리뷰 JSON 일괄 저장(비공개·owner_import·플랜 한도 적용).
- `app/dashboard/import/page.tsx` — 캡처 안내 → 인식 → 신뢰도 배지 확인 그리드 → "전부 추가".
- 진입점: `/dashboard/reviews`의 "한 번에 가져오기" 버튼.

검증: 타입체크·프로덕션 빌드 통과, curl E2E(추출 3건 → 저장 2건 → 플랜 remaining 98) 성공.

**배포 전 필수:** 프로덕션에 `ANTHROPIC_API_KEY` 세팅(없으면 503). 비용 민감하면 `ANTHROPIC_VISION_MODEL=claude-haiku-4-5`로 시작.

남은 개선(후순위): 짧은 화면 녹화 → 프레임 추출 파싱(v2), 기존 Google Vision 단건 경로와의 통합/정리.

### Step 2 (구글 있는 유저): 공식 API 딸깍
- Google Business Profile API로 owner OAuth → 진짜 URL/계정 연결 한 번에 리뷰 이관. (단 API 승인에 수주~수개월, Places API는 5개 제한 주의.)

### Step 3 (표시): 요약+출처링크+집계수치 렌더 모드
- 공개 프로필에서 외부 플랫폼 리뷰는 요약+딥링크, 평점은 집계로.

### 하지 말 것
- 서버 사이드 네이버 크롤러 (F)
- 브라우저 확장 (E) — 첫 매출 전엔 과투자

## 6. 리서치 신뢰도 메모

- 법률·반자동화 축: 다중 에이전트 + 판례 실측으로 **high confidence**.
- 검증(적대적) 에이전트 2건은 API 오류로 미완 → 위 판례(사람인/잡코리아, 야놀자/여기어때)는 널리 알려진 실제 확정판결이라 재확인 시에도 결론 불변으로 판단. 단, **실제 기능 착수 전 IP/IT 전문 변호사 1회 자문 권장**(특히 "집계 수치 표시"와 "요약 발췌" 경계).
