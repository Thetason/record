# Re:cord Development Roadmap

기준 날짜: 2026-03-27  
목적: 앞으로 무엇을 개발해야 하는지, 무엇은 아직 개발하지 말아야 하는지 현재 전략 기준으로 정리한 로드맵

## 1. 현재 단계

지금은 `기능을 더 많이 만드는 단계`가 아니라 `리포지셔닝 후반부를 닫는 단계`다.

2026-04-02 업데이트:

- 대표 후기 `feature` 개념이 데이터 모델, 공개 프로필, 대시보드 리뷰 관리에 연결되었다.
- 공개 프로필과 홈 데모는 대표 후기 우선 정렬 기준을 따른다.
- 다음 제품 초점은 `프로필 편집 단순화`와 `공유 화면의 전송 행동 강화`다.

해야 할 일은 크게 네 가지다.

1. 공개 프로필을 더 설득력 있게 만든다.
2. 사용자가 실제로 링크를 보내기까지 걸리는 시간을 줄인다.
3. 운영/배포 기준을 production-ready 수준으로 닫는다.
4. 전략과 충돌하는 오래된 축을 확장하지 않고 정리한다.

## 2. 다음 개발 우선순위

### Priority 1. 대표 후기 개념을 제품에 넣기

현재 전략상 가장 중요한데 아직 제품 구조로 충분히 표현되지 않은 부분이다.

현재 상태:

- 서버/DB 기준은 구현됨
- 공개 프로필 우선 노출 연결됨
- 대시보드 대표 후기 선택 UX 연결됨
- 남은 보완은 설명 카피와 추가 미리보기 polish 정도다

해야 할 것:

- 대표 후기 `pin/feature` 개념 추가
- 공개 프로필에서 대표 후기 우선 노출
- 대시보드에서 대표 후기 선정 UX 제공
- 공유 페이지에서 대표 후기 기준으로 미리보기 정리

완료 기준:

- 사용자가 “이 3개를 먼저 보여주겠다”를 명시적으로 설정할 수 있다.
- 공개 프로필이 단순 최신순/점수순이 아니라 의도된 설득 구조를 가진다.

### Priority 2. 프로필 편집 플로우를 더 짧고 선명하게 만들기

지금은 기능은 있지만, 사용자가 무엇부터 채워야 하는지 제품이 충분히 강하게 안내하지 않는다.

해야 할 것:

- 프로필 편집의 입력 순서를 재정렬
- 소개 문장, CTA, 작업 사진, 대표 후기 중심으로 압축
- 덜 중요한 설정은 숨김 또는 후순위로 이동
- 공개 여부 토글과 미리보기 연결 강화

완료 기준:

- 신규 사용자가 “프로필 공개까지” 도달하는 데 헤매지 않는다.
- 메인 설정 화면이 신뢰 형성에 필요한 항목만 강조한다.

### Priority 3. 공유 경험을 제품의 중심 행동으로 강화하기

현재 가치의 핵심은 저장이 아니라 `보내기`다.

현재 상태:

- 링크 복사, 소개 멘트 복사, 카카오 공유, 인스타/스레드용 복사가 이미 제공된다.
- 화면 구조도 전송 행동 중심으로 어느 정도 정리되어 있다.
- 다음 개선은 “공유 전에 무엇을 점검해야 하는지”를 더 짧게 보여주는 일이다.

해야 할 것:

- 공유 페이지에서 가장 자주 쓰는 행동만 상단 배치
- 카카오/문자/인스타 프로필 활용 문구 더 명확히
- 프로필 링크 복사 후 후속 행동 가이드 강화
- QR/임베드/부가 옵션은 후순위 정리

완료 기준:

- 사용자가 링크를 복사하고 바로 보내는 행동이 더 짧아진다.
- 공유 화면이 “부가 기능 모음”이 아니라 “전송 행동” 중심이 된다.

### Priority 4. 직접 후기 요청 루프를 더 명확히 만들기

이 기능은 현재 전략과 잘 맞지만, 메인 제품 언어로 더 선명하게 다듬을 여지가 있다.

해야 할 것:

- 요청 링크 생성과 사용법 안내 강화
- 제출 후 pending -> 검토 -> 공개 흐름의 상태 표현 개선
- 스팸/오남용 방어 보완
- “직접 받은 후기”와 “기존 플랫폼 후기”의 차이 설명 정리

완료 기준:

- 사용자가 새 후기를 계속 쌓는 루프를 자연스럽게 이해한다.
- 운영자가 검토해야 하는 포인트가 분명하다.

### Priority 5. production readiness 닫기

이건 기능 개발만큼 중요하다. 로컬에서 잘 돌아가는 것과 실제 운영 가능한 것은 다르다.

해야 할 것:

- production env 변수 기준 확정
- DB baseline 정리
- remote smoke 루틴 정착
- health check / auth / checkout / webhook 최종 점검
- OCR on/off 런칭 기준 결정

완료 기준:

- `DEPLOY_STATUS.md`의 blocker가 닫힌다.
- preview/prod에서 반복 가능한 배포 절차가 생긴다.

### Priority 6. 플랜/결제/스키마 기준 하나로 정리

이건 사용자 기능보다 내부 안정성을 위해 중요하다.

해야 할 것:

- 플랜 제한 로직의 기준 파일 하나로 정리
- 현재 사용할 결제 provider 기준 고정
- 중복 webhook/legacy payment 흐름 정리
- dev/prod Prisma schema 이중 기준 정리 계획 수립

완료 기준:

- 가격/구독/한도 정책을 한 군데서 설명할 수 있다.
- 운영 중 플랜/결제 drift 위험이 줄어든다.

## 3. 지금 당장 개발하지 않을 것

아래는 지금 확장하지 않는다.

- broad vertical 동시 확장
- 고급 통계 확장
- 과도한 커스터마이징
- Reboot 축 확장
- AI 요약/감정 분석
- 팀/조직 협업 고도화
- 공개 API/Zapier/익스텐션

이 항목들은 필요가 없어서가 아니라, `지금 제품의 선명도`를 흐리기 때문이다.

## 4. 권장 실행 순서

### Sprint A. 제품 선명도

1. 대표 후기 기능 추가
2. 프로필 편집 화면 단순화
3. 공유 화면 재정렬

### Sprint B. 활성화 루프

1. 직접 후기 요청 플로우 정리
2. migration concierge와 셀프서브 경계 정리
3. 헤어디자이너 데모/예시 개선

### Sprint C. 운영 안정화

1. env / preflight / smoke / baseline 정리
2. 결제/구독 기준 정리
3. 배포 체크리스트 실제 운영 기준으로 잠그기

## 5. 다음 마일스톤의 정의

다음 마일스톤은 아래 문장으로 정의한다.

`헤어디자이너가 30분 안에 공개 프로필을 열고, 대표 후기 3개를 고르고, 고객에게 링크를 보내고, 이후 직접 후기 요청으로 자산을 계속 쌓을 수 있다.`

이 문장에 직접 기여하지 않는 작업은 우선순위를 낮춘다.
